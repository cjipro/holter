### 3. Architecture spine for Value + Risk methodologies (PULSE-99..104, HOL-9, HOL-10)

Long design discussion crystallised three architectural moves:

**Canvas slots have three classes** — declared (author writes, validator checks), computed (engine derives at runtime, must be reproducible), attached (organizational fact, lives outside the pack). Value and Risk are *computed*, not declared — derived from data + inference + methodology, not stored as fields.

**Two new Pulse modules emerge as methodology peers of `pulse/frictionbench/`:**
- `pulse/value/` — computes Value tier from friction severity × affected population × frequency × cohort vulnerability × counterfactual baseline.
- `pulse/risk/` — computes Risk tier from friction signature + regulatory taxonomy match (FCA / Consumer Duty / EU AI Act) + bank-policy thresholds + Chronicle precedent matches.

**Risk × Value 2×2 maps to CLARK-style action tiers** — `ACUTE / REGULATORY-FLAG / COMMERCIAL-OPPORTUNITY / WATCH / NOMINAL`. The load-bearing cell is `REGULATORY-FLAG`: surfaces friction patterns that look ignorable on commercials but carry duty-of-care exposure ("not just a value question, but a regulatory flag").

Decisions locked:
- Bank-internal policies via per-deployment `bank_policy.yaml` config (peer of `taq_contract.yaml`)
- Precedent library as curated Chronicle (parallel to MIL's CHR-001..004)
- Reporting unit: tier-words (not numeric scores) in the briefing surface
- Project split rule: logical framework → PULSE; design / surface → HOL
